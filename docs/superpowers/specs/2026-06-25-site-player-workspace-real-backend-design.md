# 参赛选手端工作台真实后端化设计

## 目标

本次只处理参赛选手端工作台的真实数据边界：队伍、成员、职责、作品提交状态都以后端返回为准，不再用浏览器本地存储伪造“队长”“职责”“作品草稿”。

## 范围

- 保留现有数据库结构，不新增表，不迁移 `advisor` 字段。
- 前端队伍工作台从 `/api/site/bootstrap` 的 `teams` 和 `works` 渲染。
- 作品按钮文案从“保存草稿”改为“提交作品”，因为当前后端 `/api/work/submit` 会写入 `submitted` 状态。
- 作品表单未提交前只是页面临时输入；刷新后以数据库中已提交作品为准。
- 队长权限以后端真实用户为准：当前登录用户必须能在队伍成员或队长槽位中匹配，不能再默认伪装成队长。
- 提交成功后重新拉取 `/api/site/bootstrap`，页面显示真实提交 / 审核状态。

## 不做

- 不做数据库字段迁移。
- 不做文件上传。
- 不做管理员审核 UI 新功能。
- 不做离线草稿保存。

## 关键数据来源

- `server/siteStateService.js`：组装官网初始化状态。
- `server/index.js`：处理队伍和作品写接口。
- `server/teamRepository.js`：保存队伍成员和职责。
- `server/worksRepository.js`：保存作品提交和审核状态。
- `src/site.js`：渲染参赛选手端组队页和队伍工作台。

## 验证方式

- 单元测试必须证明前端不再读取 `WORK_DRAFT_KEY`、`TEAM_NAME_KEY`、`WORKSPACE_META_KEY` 来决定工作台数据。
- 单元测试必须证明按钮文案和提交行为是“提交作品”，不是“保存草稿”。
- 服务端测试必须继续证明写接口以 session 用户为准。
- 全量 `npm test` 必须通过。

## 追加：选手职责认领

用户确认采用“选手自己认领队长 / 队员职责”的方案。实现边界如下：

- 认领入口只出现在已加入队伍的参赛选手工作台。
- 可认领职责固定为：队长、业务洞察、AI 开发、产品设计、路演运营。
- 点击认领后调用后端 `/api/team/claim-role`，写入真实队伍成员数据。
- 同一职责只能由一个成员占用；如果已有其他成员占用，后端返回冲突错误。
- 认领“队长”后，当前用户才拥有作品提交权限。
- 前端不保存职责到 localStorage，不伪造队长状态。
