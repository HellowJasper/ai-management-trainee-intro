# 阶段沟通总结（2026-06-26）

## 本阶段结论

1. 评委评分页需要以五个评分维度为核心：创新性、技术实现、业务价值、可行性、演示表现。评分控件应保持一列一维度，支持手动输入，移动端优先保证输入框和列宽不挤压。
2. 成员维护需要同时支持“从已登录用户选择”和“手动填写”。手动填写的成员也必须能落到后端队伍成员数据里，而不是只停留在表单 UI。
3. 数据与投票管理页的信息密度过高，已经拆成更清晰的管理入口。内容管理上方适合放内嵌小导航，把业务场景大屏内容、星锐档案、内容数据入口拆开切换。
4. 排行榜不是直接读取某个单项票数，而是依赖后端投票结果、专家评分和管理员发布结果的状态。专家评分、投票开放、结果发布是三条不同流程，需要在页面文案和按钮状态上明确边界。
5. 组队锁定后，普通参赛者不应再加入、退出或认领职责；管理员仍保留维护能力。锁定的含义是“普通端 roster 只读”，不是“管理员不可编辑”。
6. 大屏路演和颁奖页需要更强的仪式感，但动态效果应服务于信息，不应影响计时器、队伍名单、冠军信息的可读性。
7. 首页任务倒计时应由管理员端配置控制。用户端不能再用写死的 2 小时作为主要显示来源，否则会出现刷新后重置、管理员控制不生效的问题。

## 关键问题复盘

1. 初期评分控件出现“一个维度两个进度条”和移动端挤压，根因是 UI 表达层没有把“滑动输入”和“手动输入”统一成同一个数据源。
2. 投票取消报错暴露了状态边界问题：前端提示、后端投票窗口、用户当前投票状态必须保持一致，否则用户会看到“我点了按钮但流程不允许”的割裂体验。
3. 排行榜为空或分数不一致的问题，本质是数据链路没有在 UI 上解释清楚：作品票数、专家评分、最终综合分不是同一个字段，也不是同一个发布时机。
4. 组队锁定后仍可操作的问题，是前端按钮状态和后端权限规则没有完全对齐；后续类似操作都应该前后端双重限制。
5. 倒计时一直重置的问题，是用户端读取公开状态时又额外请求了管理员状态，并且部分阶段只使用硬编码 fallback，导致后台配置没有成为唯一可信来源。

## 术语说明

- Bootstrap：页面初始化时一次性拿到的公开数据包，来自 `/api/site/bootstrap`，包含当前阶段、计时器、队伍、作品、投票和结果状态。
- Timer：计时器状态，主要字段是 `durationMs` 和 `startedAt`。
- `durationMs`：倒计时总时长，单位是毫秒。比如 129600000 表示 36 小时。
- `startedAt`：倒计时启动时间。为空时表示管理员只配置了时长但还没有正式启动。
- Fallback：兜底值。只有后端没有给有效计时器时，前端才应该使用兜底时间。
- Locked team：锁定队伍。普通参赛者不能再更改成员或职责，管理员仍可维护。
- Published snapshot：已发布结果快照。管理员发布后形成固定结果，排行榜和颁奖页应优先读取它。

## 后续建议

1. 为关键流程建立“状态说明卡”：评分、投票、发布、锁定、计时器分别说明当前状态和下一步动作。
2. 所有用户端写操作都保持同一策略：按钮先检查本地最新状态，提交后以后端结果为准，失败时刷新状态并给出中文原因。
3. 管理员端的每个控制按钮都应对应一个明确后端接口，并在文档里标注数据表或 JSON repository 的落点。
4. 大屏动态效果建议继续控制在背景层、入场层、数字动效层，不直接干扰核心数据布局。
